Here’s the current landscape and a plan for extending it with X‑13 seasonal adjustment and a D3 time‑series view:

 ────────────────────────────────────────────────────────────────────────────────

 ### 1. Framework pattern (from /memory/*.md)

 - Goals: build a BLS visualization platform that combines rich UI (histograms, stats panels) with rigorous statistical tooling (PQDE, KDE, future multi-dataset support).
 - Architecture: data registry → transformation/statistics engines → visualization layer, all fed from BLS/Census data sources.
 - Roadmap: after the OEWS histogram work come diagnostics, geographic rollouts, then time-series analytics with decomposition/seasonality.

 This reinforces that seasonal adjustment should slot into the “Statistics Engine” bucket, feeding finished series to the D3 visualization layer.

 ────────────────────────────────────────────────────────────────────────────────

 ### 2. Existing BLS time-series fetch (src/ui/oe-drilldown.html)

 - The “Fetch Time Series” button posts to /ui/api/bls, sending seriesid, startyear, endyear. It expects JSON shaped like the BLS public API response and caches each series under STATE.cache.
 - Returned points are sorted and rendered by the D3 code in renderTimeseries, which already draws a line/area chart in the dark UI theme.
 - Issues to fix when we wire real data:
     - The request payload has a typo ('2020/ui'); it should be a valid end year.
     - No API key is included; we need to pass it from the environment.
     - Console debug artifacts (console.log('series ID ', seriesId)) should be cleaned up.

 ────────────────────────────────────────────────────────────────────────────────

 ### 3. Credentials

 - data/.env holds both BLS_API_KEY and CENSUS_API_KEY.
 - Whatever server endpoint or Python script we use must source those values rather than hardcoding.

 ────────────────────────────────────────────────────────────────────────────────

 ### 4. X‑13 resources

 - Executable: C:\Program Files\x13as\x13as.exe.
 - Documentation PDFs: docx13as.pdf (full manual) and qrefx13aspc.pdf (quick spec reference) live in the same directory; use them if we need detailed spec syntax or diagnostic meanings.

 ────────────────────────────────────────────────────────────────────────────────

 ### 5. End-to-end plan

 1. Fetch raw NSA series
     - Use execute_python (or a small Node/Express handler) to call the BLS API:
       ```python
         import os, requests, json
         payload = {
             "seriesid": ["OEUN000000010000000000001"],
             "startyear": "2015",
             "endyear": "2024",
             "registrationkey": os.environ["BLS_API_KEY"]
         }
         resp = requests.post("https://api.bls.gov/publicAPI/v2/timeseries/data/", json=payload)
       ```
     - Extract the monthly values (value, year, period) and ensure at least 36 observations.
 2. Run X‑13 in execute_python
     - Write a .spc and call C:/Program Files/x13as/x13as.exe (pass the basename, not .spc, and run with cwd=tempdir).
     - Parse .d11 (seasonally adjusted), .d10 (trend), .d8 (irregular) as needed.
     - Return clean JSON:
       ```json
         {
           "series": "OEUN…",
           "seasonally_adjusted": [{"date": "2015-01", "value": 123.4}, …],
           "trend": […],
           "metadata": {"spec": "x11+log", "last_run": "2026-04-27T16:23Z"}
         }
       ```
 3. Persist/cache
     - Optional: write outputs to data/x13/<series>.json so the UI (or API route /ui/api/x13) can serve cached results without rerunning X‑13 every time.
 4. Expose to the front end
     - Extend /ui/api/bls (or add /ui/api/x13) so the button fetches the seasonally adjusted JSON, either by triggering the Python pipeline on demand or by reading cached files.
     - Include the API key in the server-side request, not in the client bundle.
 5. Visualize in D3
     - Reuse renderTimeseries but swap STATE.cache[sid] values to use the seasonally adjusted data.
     - Add toggles for raw vs SA vs trend lines, legend entries, and metadata badges (e.g., “X‑13 ARIMA-SEATS, log + X11”).
     - Keep the UI conventions: dark theme, axis labels, title, source notes (“Source: BLS OEWS via X‑13 ARIMA-SEATS”).
 6. Diagnostics (future)
     - Surface selected stats from .mcd, .udg, or .err files (e.g., M-statistics, AIC).
     - Store them in STATE.cache[sid].diagnostics and render a sidebar card.

 ────────────────────────────────────────────────────────────────────────────────

 With this pipeline we stay aligned with the existing framework: data source (BLS) → transformation (X‑13) → cached JSON → D3 visualization. The next actionable step is wiring a Python helper (or backend handler) that pulls    
 the raw series with the key from data/.env, runs X‑13, and hands the sanitized JSON back to the UI.
